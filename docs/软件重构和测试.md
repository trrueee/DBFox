DBFox 代码质量审查报告
一、项目整体理解
DBFox 是一个本地优先、AI 原生的数据库工作台桌面应用。解决的核心问题是：让数据库开发者和分析师通过可视化界面 + 自然语言 AI Agent 来管理数据源、浏览 Schema、执行 SQL、进行对话式数据分析。
技术栈：Python FastAPI 引擎 + React 19/TypeScript 前端 + Tauri 2 桌面壳。通信方式：HTTP REST + SSE 流式推送，本地 127.0.0.1:18625。
核心用户流程：

1. 配置数据源（MySQL/PostgreSQL/SQLite，支持 SSH 隧道和 SSL）
2. 同步 Schema → 浏览表结构 → 查看 ER 图
3. SQL 控制台执行查询（Monaco 编辑器）
4. AI 问数 Agent（自然语言 → SQL → 执行 → 展示结果）
核心模块：

- engine/api/ — 9 个 REST 路由模块
- engine/sql/ — Guardrail + TrustGate + Executor（SQL 安全门）
- engine/agent_core/ — 25 个文件，Agent 运行时核心
- engine/tools/ — Agent 工具注册（db, memory, semantic, environment）
- engine/policy/ — 安全策略（数据脱敏、确认机制、工具验证）
- engine/crypto.py — AES-256-GCM 密码加密
- desktop/src/ — React 前端，5 个 feature 模块

## 三、主要质量优点

1. __安全模型设计精良__：Guardrail（AST 级 SQL 静态分析）+ TrustGate（运行时安全决策）+ 数据脱敏 + 确认机制 + Origin 校验 + Token 鉴权，形成了多层纵深防御。
2. __加密存储__：数据库密码使用 AES-256-GCM 加密，密钥优先存储在 OS Keychain，有完整的迁移路径。
3. __类型系统完备__：前后端都使用严格的类型定义（Pydantic + TypeScript），API 契约清晰。
4. __错误层次清晰__：`DBFoxError` 基类 + 6 个子类，全局异常处理器兜底，不泄露内部调用栈。
5. __测试覆盖有策略__：后端 54 个测试文件，前端 22 个测试文件，包含架构守卫测试（source-scanning tests）。
6. __工具系统可扩展__：YAML 声明式工具定义 + Handler 注册 + 优先级覆盖 + 用户自定义工具源。
7. __连接池管理__：MySQL/PostgreSQL 使用 SQLAlchemy QueuePool，带 LRU 驱逐和健康检查。

四、高风险问题清单
🔴 严重 (Critical)
C1. _kill_mysql_query 中 SQL 拼接

- 位置：engine/query_registry.py:168
- 问题：cursor.execute(f"KILL QUERY {int(thread_id)}") 使用 f-string 拼接 SQL。虽然 int() 做了类型转换防止注入，但这种模式违反了参数化查询原则，且 int() 对非数字字符串会抛 ValueError 未被捕获。
- 风险：若 thread_id 来源被污染（虽然当前场景不太可能），可能导致 SQL 注入。
- 修复：改为 cursor.execute("KILL QUERY %s", (thread_id,))。
C2. guardrail_check 返回值包含内部字段 _parsed_ast
- 位置：engine/sql/guardrail.py:402
- 问题：GuardrailResult TypedDict 声明中没有 _parsed_ast 字段，但函数在 return 中返回了它。这是一个隐式的类型逃逸。
- 风险：任何将 GuardrailResult 序列化为 JSON 的代码（如 FastAPI response）都会暴露内部 AST 对象，可能导致序列化错误或信息泄露。
- 证据：_public_guardrail_result 函数专门做了_ 前缀过滤来处理这个问题，说明团队意识到了，但根因未修。
- 修复：将 _parsed_ast 从返回值中分离，通过独立参数或上下文传递。
C3. app.tsx God Component — 47 props 传递
- 位置：desktop/src/App.tsx:385行 + WorkspaceRouter 47 个 props
- 问题：所有应用状态集中在 App.tsx，通过 props 逐层传递。任何子组件的接口变更都会级联影响整个组件树。
- 风险：极高的维护成本，重构困难，容易引入 prop 不一致的 bug。
- 修复：引入 React Context 或 Zustand 等状态管理方案，按 feature 拆分 context。
🟡 高 (High)
H1. engine/sql/executor.py 超过 1000 行
- 位置：engine/sql/executor.py（1011 行）
- 问题：单个文件包含连接池管理、3 种数据库执行器、行处理序列化、Guardrail 集成、历史记录、EXPLAIN 等所有逻辑。
- 风险：认知负荷高，修改一个数据库方言的逻辑可能影响其他方言。
H2. datasource.py 超过 585 行
- 位置：engine/datasource.py
- 问题：SSH 隧道管理、连接测试、SSL 参数构建、连接参数获取全部混在一个文件。
- 风险：TunnelManager 是全局单例，线程安全依赖单把锁，SSH 隧道泄漏风险。
H3. 前端重复的 executeSql 函数
- 位置：desktop/src/features/engine/engineApi.ts 和 desktop/src/lib/api/query.ts
- 问题：两个同名函数执行相同操作，调用者不同。
- 风险：行为不一致时难以排查。
H4. ConfirmationManager 使用内存存储
- 位置：engine/policy/confirmation.py:29
- 问题：_confirmations dict 存储在内存中，进程重启后丢失。如果用户创建确认后引擎重启，确认令牌永久失效。
- 风险：用户体验问题，但因为是本地应用，重启概率低，实际影响有限。
H5. 全局单例 _registry 和 QUERY_REGISTRY
- 位置：engine/agent_core/handler_registry.py:113、engine/query_registry.py:173
- 问题：多个模块级全局单例，测试隔离困难，热重载可能产生状态残留。
🟢 中 (Medium)
M1. _datasource_to_dict 手动序列化
- 位置：engine/api/datasources.py:40-78
- 问题：手动将 ORM 对象转为 dict，字段增减时容易遗漏。
- 修复：使用 Pydantic schema 或 sqlalchemy Serializer。
M2. DataRedactor 脱敏规则过于简单
- 位置：engine/policy/redactor.py
- 问题：手机号正则只覆盖中国大陆格式，信用卡正则可能误匹配非信用卡数字。
M3. main.py 开发模式自动写入 .env.local
- 位置：engine/main.py:97-111
- 问题：引擎启动时自动覆盖前端配置文件，若多实例运行会互相覆盖。
M4. 前端核心类型定义在 mock/dbfoxMock.ts
- 位置：desktop/src/mock/dbfoxMock.ts
- 问题：WorkspaceTab 等核心类型定义在名为 "mock" 的文件中，命名误导。
M5. guardrail.py:367 在函数内部 import logging
- 位置：engine/sql/guardrail.py:367
- 问题：函数内重复导入，应使用模块级 logger。
M6. executor.py 返回值同时包含 camelCase 和 snake_case
- 位置：engine/sql/executor.py:729-741
- 问题：connectMs 和 connect_ms 同时存在于返回 dict，为了兼容前端和后端测试。
- 风险：字段名不一致导致混淆。
